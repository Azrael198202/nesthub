# NestHub 智能体分类与工作流框架

## 1. 目标

本框架用于统一 NestHub 中不同类型智能体的职责边界，并说明它们如何在工作流中协同完成用户请求。

目标不是把所有能力都做成智能体，而是明确：

1. 哪些能力属于知识沉淀。
2. 哪些能力属于动作执行。
3. 哪些能力属于内容分析。
4. 哪些能力属于工作流编排。
5. 工具、分析模块、I/O 操作如何被工作流调度。

---

## 1.1 系统总原则

NestHub 的智能体体系遵循一个强约束：

1. 所有智能体的创建由 `nesthub_runtime` 系统完成。
2. 所有工作流的编排与执行由 `nesthub_runtime` 系统完成。
3. 生成过程和执行过程中出现的问题，优先由 `nesthub_runtime` 系统自行识别、重试、降级、补全或修复。
4. 用户负责提出目标，不负责手工拼装智能体、手工拼装工作流、手工补运行逻辑。

因此，用户面向的是“目标”，而不是“系统内部搭建过程”。

NestHub 需要自己承担：

- 需求理解
- 智能体类型判断
- 智能体配置生成
- 工作流拆解
- 能力路由
- 工具调度
- 中间问题处理
- 最终结果交付

这条原则适用于：

- 信息类智能体创建
- 执行类智能体创建
- 分析类智能体创建
- 工作流编排类智能体创建
- 运行时问题处理与补全

## 2. 智能体分类

### 2.1 信息收集类智能体

职责：

- 收集结构化信息。
- 持久化为知识记录。
- 为执行类与工作流类智能体提供上下文。

典型实例：

- 家庭成员信息智能体
- 日程信息智能体
- 外部联系人信息智能体
- 地点信息智能体
- 资产/物品信息智能体
- 偏好与规则信息智能体

框架属性：

- `agent_class = information`
- `agent_layer = knowledge`

---

### 2.2 执行类智能体

职责：

- 接收用户指令。
- 读取知识类智能体提供的上下文。
- 调用工具或外部接口完成动作。

典型实例：

- 提醒执行智能体
- 消息发送智能体
- 文档生成执行智能体
- 任务安排执行智能体

框架属性：

- `agent_class = execution`
- `agent_layer = execution`

---

### 2.3 分析类智能体

职责：

- 解释、总结、比较、抽取内容。
- 不直接承担长期知识建库。
- 不直接承担外部动作执行。

典型实例：

- 内容总结智能体
- 风险检查智能体
- 文档分析智能体
- 结构化提取智能体

框架属性：

- `agent_class = analysis`
- `agent_layer = analysis`

---

### 2.4 工作流编排类智能体

职责：

- 将知识类、执行类、分析类、工具、I/O 操作组合成一个可执行流程。
- 根据任务目标拆分步骤并协调依赖。

典型实例：

- 出行准备工作流
- 家庭事务处理工作流
- 文档整理与投递工作流

框架属性：

- `agent_class = orchestration`
- `agent_layer = workflow`

---

## 3. 联系人建模边界

### 3.1 家庭成员信息智能体

只记录家庭内部成员，包括：

- 爸爸、妈妈、孩子、老人等
- 联系方式、角色、币种、提醒偏好、消费统计参与状态

### 3.2 外部联系人信息智能体

只记录家庭外部联系人，包括：

- 老师
- 同事
- 医生
- 邻居
- 物业
- 维修人员
- 公司对接人

因此，联系人智能体不是家庭成员联系方式的重复表，而是另一个对象域。

---

## 4. 工作流组合模型

一个完整工作流由以下组件构成：

1. 知识类智能体：提供背景知识。
2. 执行类智能体：完成任务动作。
3. 分析模块：生成中间分析结果。
4. 工具：生成文档、读写文件、调用接口等。
5. I/O 操作：保存结果、发送消息、投递文档。
6. 工作流编排器：协调整体执行。

示例：

用户请求：

> 根据爸爸下周出差大阪的日程，整理需要联系的外部联系人，生成提醒文档，并在前一天提醒我。

推荐编排：

1. 从日程信息智能体读取出差安排。
2. 从家庭成员信息智能体读取爸爸资料。
3. 从外部联系人智能体读取大阪相关联系人。
4. 用分析模块整理提醒重点。
5. 用文档工具生成提醒文档。
6. 用提醒执行智能体创建提醒任务。

## 6. 当前实现策略

当前策略是：

- 信息类智能体先落地。
- 通过需求信号自动推导 `directory` / `timeline` / `generic information`。
- 家庭成员与外部联系人同属 `entity_directory`，但对象域不同。
- 日程属于 `structured_timeline`，支持直接结构化入库。
- 工作流执行后，`nesthub_runtime` 会先做结果验收；若出现失败步骤或结果未满足输出要求，则由系统自动生成补救工作流并重跑一次。

### 6.1 运行时自修复闭环

当前 `nesthub_runtime` 已经具备一条最小可用的自修复闭环：

1. 执行当前 workflow。
2. 评估结果是否满足 `output_requirements`。
3. 检查是否存在失败步骤。
4. 若不满足要求，则自动生成 repair workflow。
5. 系统自行追加补救步骤，例如：
  - 重试失败步骤
  - 注入分析步骤
  - 注入文档生成步骤
  - 注入结果持久化步骤
6. 再执行一次补救 workflow。

这条链的目标是让 `nesthub_runtime` 自己承担“发现问题并继续作”的责任，而不是把不完整结果直接交给用户。
